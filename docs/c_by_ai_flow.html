Critical bug: the entire meal plan from GET /meal-plan/{device_id} was silently never loading. UI only showed data from the polling stream. After cold restart, tracker was blank.

2 fitness metrics never set after cold restart

❌ BEFORE: only updated fitnessMetrics if it was already non-null
✅ AFTER: always reads user_info → { bmi, bmi_overview, bmr, tdee, body_fat } from plan response
After app restart + recovery via check-meals → meal-plan, the tracker showed 0 for BMI, TDEE, body fat because fitnessMetrics was never populated.

3 session_id not saved from plan response

❌ BEFORE: session_id only saved from POST /generate-meals/start
✅ AFTER: GET /meal-plan response includes session_id → saved to SharedPrefs
After cold restart + recovery, approve and schedule-delivery calls would fail with null session_id. Now it's always available.

4 recoverSession used wrong resume endpoint

❌ BEFORE: POST /generate-meals/status (wrong method + endpoint)
✅ AFTER: GET /session/{session_id}/status (correct per API docs)
Also now handles "pending" and "partially_completed" statuses from the resume response.

5 partially_completed caused infinite polling

❌ BEFORE: only status == "completed" stopped the polling timer
✅ AFTER: status == "completed" || status == "partially_completed" → done
API docs say: partially_completed = all days done with minor issue on one day. Treat same as completed. Previously the app would poll forever and show a timeout error.

6 generateMeals looked for user_metrics in start response

❌ BEFORE: data['user_metrics'] from POST /start response (never present)
✅ AFTER: Metrics come exclusively from GET /meal-plan → user_info
The start API returns only { success, session_id, message }. user_metrics was always null, so fitnessMetrics was always zero. Now set correctly when plan loads.

+ Pre-flight check before generation

❌ BEFORE: CByAiCalculatingScreen always called generateMeals regardless
✅ AFTER: checkForExistingPlan() runs first → skip to Tracker if plan exists
New gate: even if user navigates through Profile → Target → Continue, the app checks check-meals before firing POST /generate-meals/start. If a plan already exists (e.g. after subscription + reinstall), user goes directly to tracker.
